# 数据流水线模块设计

> 关联：[DESIGN.md](../DESIGN.md) · [比赛与记录](./03-比赛与记录模块.md) · [Rating](./05-Rating与榜单模块.md)

---

## 1. 模块职责

- 定义 `data/raw` → `data/processed` → `data/public` 的**格式、目录与处理顺序**
- 实现「**只处理、不爬取**」：数据由人工/外部系统放入 `raw/`
- 提供 CLI：`xcpc-data import | process | export | update`

## 3. 处理阶段

```
┌─────────┐    import     ┌────────────┐    process     ┌────────────────┐
│  raw/   │ ────────────▶ │ 校验+匹配   │ ─────────────▶ │  processed/    │
└─────────┘               │ 选手/队伍   │                │  各 JSON 表     │
                          └────────────┘                └───────┬────────┘
                                                                │
                     ┌──────────────────────────────────────────┤
                     │  rating (算法插件)                        │
                     ▼                                          │
              ┌──────────────┐    export    ┌──────────────────▼──┐
              │ rating 状态   │ ──────────▶ │  public/            │
              │ 快照/历史     │             │  榜单+详情+目录       │
              └──────────────┘             └─────────────────────┘
```

| 阶段 | 命令 | 输入 | 输出 |
|------|------|------|------|
| import | `xcpc-data import` | `raw/**` | 更新 `processed/*` 部分表 |
| process | `xcpc-data process` | `processed` 基础表 | `rating_events.json` |
| rating | `xcpc-data rating` | `rating_events` + 选手 | 内存/临时 rating 状态 |
| export | `xcpc-data export` | 全部 processed + rating | `public/**` |
| update | `xcpc-data update` | 以上全部 | 等价于 import→process→rating→export |

## 4. 原始数据（raw）约定

### 4.1 原则

- 一文件一场次或一批快照，文件名建议带日期：`{id}.json` 或 `{platform}_{date}.csv`
- **不修改 raw**：重跑 import 只读 raw，便于复现
- 支持格式：选手名册 **JSON**；OJ 等导出可为 CSV / JSON

### 4.2 选手名册 `raw/players/roster.json`

完整样例见 **`data/raw/`**（含选手名册、正式赛、训练赛单打/双打/三人）。

```json
[
  {
    "id": "p20230042",
    "name": "张三",
    "grade": 2023,
    "oj_accounts": [
      { "platform": "codeforces", "handle": "zhangsan_cf" }
    ]
  }
]
```

### 4.3 增量导入

`import` 支持：

- **merge**：按主键合并，新记录覆盖旧字段
- **replace**：整表替换（用于名册全量更新）

配置见 `backend/data/config.yaml`：

```yaml
import:
  players: merge
  formal_contests: merge
  default: merge

paths:
  raw: data/raw
  processed: data/processed
  public: data/public
  schemas: data/schemas
  weights: data/config/contest_weights.yaml
```

权重表见 [07-比赛权重](./07-比赛权重.md)。

## 5. 中间层（processed）

### 5.1 表之间引用

```
players ◀── teams.members[]
players ◀── standings_*.player_id / teams ──▶ teams
players ◀── oj_* .player_id
contests_* ◀── standings_*.contest_id
```

### 5.2 是否使用 SQLite

| 条件 | 建议 |
|------|------|
| MVP、表少、全量重算 | **仅 JSON** 文件 |
| 表 > 10、需 SQL 聚合、增量复杂 | 可选 SQLite 作 processed 缓存，**export 仍为 public JSON** |

当前设计以 JSON 为准，SQLite 为可选优化。

### 5.3 `rating_events.json`

归一化后的事件列表（见 [03-比赛与记录模块](./03-比赛与记录模块.md)），按 `date` 排序，供 Rating 引擎消费。

## 6. 导出层（public）

### 6.1 `meta.json`

```json
{
  "built_at": "2026-06-29T08:00:00Z",
  "data_version": 12,
  "git_sha": "abc1234",
  "rating_algorithm": "placeholder_v0",
  "counts": {
    "players": 156,
    "teams": 48,
    "formal_contests": 8,
    "training_contests": 42
  },
  "rating_modes": ["formal_only", "all"],
  "period_types": ["career", "competition_year", "season"]
}
```

### 6.2 `catalogs/competition_years.json`

```json
[
  {
    "id": 2025,
    "label": "2025赛年",
    "start": "2025-09-01",
    "end": "2026-08-31"
  }
]
```

### 6.3 `catalogs/seasons.json`

```json
[
  {
    "id": "2025-秋学期",
    "competition_year": 2025,
    "kind": "autumn_semester",
    "label": "2025 秋季学期",
    "start": "2025-09-01",
    "end": "2026-01-31"
  },
  {
    "id": "2025-寒假",
    "competition_year": 2025,
    "kind": "winter_break",
    "label": "2025 寒假",
    "start": "2026-02-01",
    "end": "2026-02-28"
  }
]
```

日期边界在 `config.yaml` 的 `calendar` 段配置，processed 阶段写入 catalogs。

### 6.4 榜单文件命名

```
public/ratings/{mode}/{period_type}_{period_id}.json
```

示例：

- `ratings/formal_only/career.json`
- `ratings/formal_only/year_2025.json`
- `ratings/formal_only/season_2025-春学期.json`
- `ratings/all/career.json`

`mode`：`formal_only` | `all`

### 6.5 `public/index.json`

前端启动时可选读取，避免硬编码路径：

```json
{
  "ratings": {
    "formal_only": {
      "career": "/data/ratings/formal_only/career.json",
      "years": ["/data/ratings/formal_only/year_2025.json"],
      "seasons": ["/data/ratings/formal_only/season_2025-春学期.json"]
    },
    "all": { }
  },
  "default_mode": "all",
  "default_period": "career"
}
```

## 7. Schema 与类型

`data/schemas/` 为每类 public/processed 文件维护 JSON Schema。

工作流：

1. Python export 前 `jsonschema.validate`
2. 可选：`datamodel-code-generator` 生成 `frontend/src/types/data/*.ts`

## 8. 原子导出

```python
# 伪代码
def export_public(out_dir: Path):
    staging = out_dir / ".staging"
    write_all(staging)
    validate_schemas(staging)
    atomic_replace(staging, out_dir)
```

失败时保留上一轮 `public/`，并写 `meta.json` 的 `last_error`（可选）。

## 10. 与「不爬取」的边界

| 做 | 不做 |
|----|------|
| 解析 raw 下人工放置的文件 | 请求 OJ 网站 API |
| 校验、合并、去重 | 定时爬比赛榜单 |
| 导出静态 JSON | 运行时拉取外部数据 |

外部系统（办赛平台、OJ 导出脚本）负责把文件放进 `raw/` 或通过 CI 上传。
