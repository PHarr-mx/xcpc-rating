# 比赛与记录模块设计

> 关联：[需求文档](../需求文档.txt) · [选手](./01-选手模块.md) · [队伍](./02-队伍模块.md) · [Rating](./06-Rating计算模块.md) · [比赛权重](./07-比赛权重.md)

---

## 1. 模块职责

定义三类**可计入 Rating 的数据来源**及其原始/中间/导出结构。  
**本系统不爬取数据**，只接收人工或外部导出的文件，完成校验、关联选手/队伍、写入 `processed/`。

---

## 2. 数据类型总览

```
                    ┌─────────────────────────────────────┐
                    │           选手 Rating 输入           │
                    └─────────────────────────────────────┘
         ┌──────────────────┬──────────────────┬──────────────────┐
         ▼                  ▼                  ▼
   ┌───────────┐      ┌───────────┐      ┌───────────────────────┐
   │ 正式比赛   │      │ 校内训练赛 │      │ OJ 记录                │
   │ formal    │      │ training  │      │ ├── oj_contest (比赛)  │
   └───────────┘      └───────────┘      │ └── oj_practice (做题) │
         │                  │            └───────────────────────┘
         │                  │                  │
   仅校内队伍明细      完整排行榜          绑定 OJ 账号
   + 总队伍数          队内/个人成绩        Rating + 做题数
```

| 类型 | `source_type` | 计入「仅正式赛榜」 | 计入「全部数据榜」 | 粒度 |
|------|---------------|-------------------|-------------------|------|
| 正式比赛 | `formal` | ✅ | ✅ | 参赛单元（1～3 人）为主，映射到选手 |
| 校内训练赛 | `training` | ❌ | ✅ | 参赛单元或个人排行榜 |
| OJ 比赛 | `oj_contest` | ❌ | ✅ | 选手 + 平台 |
| OJ 做题 | `oj_practice` | ❌ | ✅ | 选手 + 平台 |

## 3. 正式比赛（formal）

### 3.1 业务规则

- 只保存**校内相关队伍**的：排名、获奖、解题数、罚时等
- 同时保存该场**参赛总队伍数** `total_teams`（含外校，用于估算难度/权重）
- 不保存外校队伍明细

### 3.2 比赛元信息 `processed/contests_formal.json`

```json
[
  {
    "id": "formal_2026_spring_invitational",
    "title": "2026 春季 ICPC 组队校赛",
    "date": "2026-04-12",
    "competition_year": 2025,
    "season": "2025-春学期",
    "contest_type": "icpc_team_school",
    "contest_type_label": "ICPC 组队校赛",
    "format": "team_xcpc",
    "total_teams": 86,
    "school_teams_count": 24,
    "rated": true,
    "weight": 60,
    "weight_source": "config",
    "source_file": "raw/formal/2026_spring_invitational.json"
  }
]
```

| 字段 | 说明 |
|------|------|
| `contest_type` | 正式赛类型枚举，**决定 Rating 权重**（见 [07-比赛权重](./07-比赛权重.md)） |
| `competition_year` | 所属赛年（9/1 划分，见 05 文档） |
| `season` | 所属赛季标签 |
| `total_teams` | 全场队伍数，**必填** |
| `rated` | 本场是否参与 Rating 计算 |
| `weight` | 整数权重，基准 100；由 `contest_type` 查表，可被 `weight_override` 覆盖 |
| `weight_source` | `config` \| `override` |

### 3.3 成绩 `processed/standings_formal.json`

```json
[
  {
    "contest_id": "formal_2026_spring_invitational",
    "team_id": "t_a1b2c3",
    "team_name": "三大队",
    "members": ["p20230042", "p20230015", "p20230101"],
    "size": 3,
    "rank": 3,
    "school_rank": 1,
    "award": "gold",
    "solved": 8,
    "penalty": 1200
  }
]
```

### 3.4 原始导入 `data/raw/formal/{contest_id}.json`（示例）

人工维护或从办赛系统导出，字段可映射：

```json
{
  "title": "2026 春季 ICPC 组队校赛",
  "date": "2026-04-12",
  "contest_type": "icpc_team_school",
  "format": "team_xcpc",
  "total_teams": 86,
  "standings": [ ]
}
```

可选 **`weight_override`**（>100 的极少数场次）及 `weight_override_reason`。  
导入时根据 `contest_type` 从 `data/config/contest_weights.yaml` 填充 `weight`。

## 4. 校内训练赛（training）

校内比赛（训练赛）在元信息上需标注 **赛制 `format`** 与 **难度档 `division`**。正式赛（第 3 节）一般为对外 `team_xcpc`，也可选填 `division`。

### 4.1 赛制（format）

三种取值，命名与「是否组队」对齐：

| `format` | 名称 | 参赛方式 | 成绩字段 |
|----------|------|----------|----------|
| `team_xcpc` | 组队 XCPC | `members` 1～3 人 | `rank`, `solved`, `penalty` |
| `solo_xcpc` | 个人 XCPC | 单人（`members` 长度 1） | `rank`, `solved`, `penalty` |
| `oi` | OI | 单人（`members` 长度 1） | `rank`, `score` |

**为何不用 `individual`？**

- 与 `team_xcpc` 对称，个人 XCPC 用 **`solo_xcpc`**（`solo` = 单人参赛），更短也更常见
- **OI 没有组队赛**，因此不叫 `solo_oi` / `team_oi`，直接 **`oi`** 即可

```json
{ "format": "solo_xcpc", "format_label": "个人 XCPC" }
{ "format": "oi", "format_label": "OI" }
```

- **组队 XCPC**：ICPC 规则；`members` 按 [02-队伍模块](./02-队伍模块.md) 识别 `team_id`
- **个人 XCPC**：每人独立榜单，ICPC 式 `solved` + `penalty`
- **OI**：每人独立，按题目分值累加 `score`；无 `solved` / `penalty`

### 4.2 难度档（division）

| `division` | 名称 | 面向对象（语义） |
|------------|------|------------------|
| `div1` | 老队员组 | 已在队、经验丰富的队员 |
| `div2` | 新队员组 | 入队较新或培养中的队员 |
| `div1+2` | 全体队员组 | 不区分新老，所有在队队员 |
| `div3` | 未入队新生组 | 尚未加入正式队伍的的新生 |

```json
{
  "division": "div2",
  "division_label": "新队员组"
}
```

**说明**：

- `division` 是**比赛标签**，导入时写在 contest 元信息上；谁可参加由办赛方控制，系统默认**信任 raw 文件**，不强制校验选手归属
- 可选：在 `backend/data/config.yaml` 配置各档规则，import 时做**警告**（如 div3 选手不应出现在校队 roster 中）
- 同一场训练可只开一个档；也可拆成多场文件（如 `…_div1.json`、 `…_div2.json`）

### 4.3 业务规则

- **完整排行榜**：本场所有参赛单元均录入（通常仅校内）
- **`weight` 由 `division` 查表**（与 `format` 无关），见 [07-比赛权重](./07-比赛权重.md)
- 默认不写在 raw 里；import 时写入 processed

### 4.4 比赛元信息 `processed/contests_training.json`

```json
[
  {
    "id": "training_2026_w12_team_xcpc_div1+2",
    "title": "第 12 周校内训练 · 组队 XCPC · 全体",
    "date": "2026-03-20",
    "competition_year": 2025,
    "season": "2025-春学期",
    "format": "team_xcpc",
    "format_label": "组队 XCPC",
    "division": "div1+2",
    "division_label": "全体队员组",
    "rated": true,
    "weight": 100,
    "weight_source": "config",
    "source_file": "raw/training/2026_w12_team_xcpc_div1+2.json"
  },
  {
    "id": "training_2026_w08_oi_div3",
    "title": "第 8 周校内训练 · OI · 未入队新生",
    "date": "2026-02-28",
    "competition_year": 2025,
    "season": "2025-春学期",
    "format": "oi",
    "format_label": "OI",
    "division": "div3",
    "division_label": "未入队新生组",
    "rated": true,
    "weight": 60,
    "weight_source": "config",
    "source_file": "raw/training/2026_w08_oi_div3.json"
  }
]
```

### 4.5 成绩 `processed/standings_training.json`

**组队 XCPC**（`members` 1～3）：

```json
{
  "contest_id": "training_2026_w12_team_xcpc_div1+2",
  "team_id": "t_a1b2c3",
  "team_name": "俩小伙",
  "members": ["p20230001", "p20230002"],
  "size": 2,
  "rank": 1,
  "solved": 5,
  "penalty": 80
}
```

**个人 XCPC（`solo_xcpc`）**：

```json
{
  "contest_id": "training_2026_w05_solo_xcpc_div2",
  "player_id": "p20230001",
  "members": ["p20230001"],
  "size": 1,
  "rank": 3,
  "solved": 4,
  "penalty": 60
}
```

**OI（`oi`）**：

```json
{
  "contest_id": "training_2026_w08_oi_div3",
  "player_id": "p20240001",
  "members": ["p20240001"],
  "size": 1,
  "rank": 2,
  "score": 280
}
```

`record_kind`（可选）：`team`（`team_xcpc`）| `solo`（`solo_xcpc` / `oi`）。

### 4.6 原始导入 `data/raw/training/{id}.json`

```json
{
  "title": "第 12 周校内训练",
  "date": "2026-03-20",
  "format": "team_xcpc",
  "division": "div1+2",
  "standings": [
    {
      "team_name": "俩小伙",
      "members": ["张三", "李四"],
      "rank": 1,
      "solved": 5,
      "penalty": 80
    }
  ]
}
```

```json
{
  "title": "第 8 周新生 OI 训练",
  "date": "2026-02-28",
  "format": "oi",
  "division": "div3",
  "standings": [
    {
      "members": ["吴九"],
      "rank": 1,
      "score": 320
    }
  ]
}
```

导入流程：解析 `format` / `division` → 校验成绩字段 → 匹配 `team_id` 或 `player_id` → 写入 processed。

## 5. OJ 记录

OJ 分两种子类型，均挂在**选手 OJ 账号**上。

### 5.1 OJ 比赛（oj_contest）

平台举办的 Rated 比赛/轮次，贡献 **OJ Rating** 变化。

`processed/oj_contests.json`：

```json
[
  {
    "id": "oj_cf_900",
    "platform": "codeforces",
    "title": "Codeforces Round 900",
    "date": "2026-05-01",
    "rated": true,
    "weight": 0.3
  }
]
```

`processed/oj_contest_results.json`：

```json
[
  {
    "event_id": "oj_cf_900",
    "player_id": "p20230042",
    "platform": "codeforces",
    "handle": "zhangsan_cf",
    "rank": 120,
    "rating_before": 1800,
    "rating_after": 1820,
    "delta": 20
  }
]
```

若导入仅有赛后 Rating，则 `rating_before` 可空，由流水线按时间序推算。

### 5.2 OJ 做题（oj_practice）

不特指某场比赛，表达**累计做题能力**：平台 Rating + 做题数量。

`processed/oj_snapshots.json`（按日/周快照，或每次导入全量）：

```json
[
  {
    "player_id": "p20230042",
    "platform": "luogu",
    "handle": "zhangsan_lg",
    "date": "2026-06-29",
    "rating": 橙名,
    "rating_numeric": 2200,
    "solve_count": 450,
    "source": "manual_import"
  }
]
```

| 字段 | 说明 |
|------|------|
| `rating` / `rating_numeric` | 平台显示名与可比较数值（各平台规则不同，processed 层保留原值 + 归一化分） |
| `solve_count` | 通过题目数（AC 数） |
| `date` | 快照日期，用于赛季截止取值 |

**做题数**与 **OJ 比赛 Rating** 可同时存在；合并策略在 Rating 模块定义。

### 5.3 原始导入目录

```
data/raw/
├── formal/
├── training/
└── oj/
    ├── contests/          # 各平台比赛结果导出
    └── snapshots/         # 各平台 rating + 做题数导出
        ├── codeforces_2026-06-29.csv
        └── luogu_2026-06-29.json
```

格式允许 CSV / JSON，由 `backend/data/import/` 适配器解析。

## 6. 统一事件模型（内部）

Rating 计算前，将异构记录转为统一 **RatingEvent**：

```json
{
  "event_id": "formal_2026_spring_invitational#t001",
  "source_type": "formal",
  "date": "2026-04-12",
  "competition_year": 2025,
  "season": "2025-春学期",
  "contest_type": "icpc_team_school",
  "contest_format": "team_xcpc",
  "player_id": "p20230042",
  "team_id": "t_a1b2c3",
  "payload": {
    "rank": 3,
    "total_teams": 86,
    "award": "gold",
    "scoring": "xcpc"
  },
  "weight": 60
}
```

- `weight` 为整数（基准 100），由 [07-比赛权重](./07-比赛权重.md) 查表写入
- 队伍赛：为每名队员生成一条事件，`payload` 相同，`player_id` 不同
- `source_type` 用于过滤「仅正式赛」榜

---

## 7. 时间标签：赛年与赛季

每条比赛/记录在 processed 阶段打上：

- `competition_year`：由 `date` 按赛年规则计算
- `season`：由 `date` 按赛季日历计算

规则详见 [06-Rating计算模块](./06-Rating计算模块.md)。  
导入文件若已带标签，以配置决定是否覆盖。

---

## 8. 处理流水线（本模块边界）

```
data/raw/{type}/**     ──import──▶  processed/contests_*.json
                                   processed/standings_*.json
                                   processed/oj_*.json
                                         │
                                         ▼
                                   normalize → rating_events.json
```

不负责：`fetch` 爬取、Rating 数值计算、前端展示。

## 9. 校验清单

| 检查 | 说明 |
|------|------|
| `date` 合法 | 必填 |
| 正式赛 `contest_type` | 必填，枚举合法，查权重表 |
| 训练赛 `format` / `division` | 必填，枚举合法；`weight` 由 division 查表 |
| `team_xcpc` 成绩含 `solved`+`penalty` | `oi` 用 `score` |
| `solo_xcpc` / `oi` 每场 `members` 长度必须为 1 | 警告或拒绝 |
| 正式赛 `total_teams` | 必填且 ≥ 本校队伍数 |
| `team_id` / `player_id` 可解析 | 否则进 unmatched |
| OJ `platform+handle` 可映射选手 | 否则进 unmatched |
| 重复导入 | 同一 `contest_id`+`team_id` 以最新导入为准 |
